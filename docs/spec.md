# Spec: 停车明细分析报告智能生成系统

## 1. 背景

本系统用于将停车场交易明细 CSV 和 Word 报告模板自动转换为一份可下载的《停车明细分析报告》。

系统需要支持用户上传数据和模板，后台异步生成报告，并在报告中包含确定性计算指标、真实图表和由智能体生成的经营观察与建议。

本项目目标不是构建完整文档平台，而是在较小范围内展示：

- 确定性计算与 LLM 推理的边界
- 基于事实的 Agent 报告规划能力
- 异步任务处理
- 可观测日志
- 可下载 Word 报告生成

## 2. 用户故事

### US-1：上传数据并创建报告任务

作为运营人员，我希望上传停车交易 CSV 和报告模板，这样系统可以为我创建一份自动分析报告。

验收要点：

- 用户可以上传 `.csv` 数据文件
- 用户可以上传 `.docx` 报告模板
- 系统返回 `job_id`
- 报告生成不阻塞 HTTP 请求

### US-2：查看生成状态

作为运营人员，我希望在提交后查看报告生成状态，这样我可以知道任务是否仍在处理中、已完成或失败。

验收要点：

- 用户可以通过 `job_id` 查询状态
- 支持 `pending / running / completed / failed`
- 页面刷新后状态仍然正确

### US-3：下载生成报告

作为运营人员，我希望在任务完成后下载 Word 报告，这样我可以直接发给业务管理者或进一步编辑。

验收要点：

- 任务完成后提供下载链接
- 下载文件格式为 `.docx`
- 未完成或失败任务不能下载成功报告

### US-4：获得基于数据的经营观察

作为停车业务管理者，我希望报告不仅展示硬指标，还能指出值得关注的经营现象和建议。

验收要点：

- 报告包含 2-3 条补充观察
- 每条观察有数据依据
- 建议面向经营管理，而不是泛泛而谈

### US-5：可选提供业务背景

作为用户，我希望可以上传领域知识或填写处理说明，这样系统生成的观察能更符合本停车场的业务背景。

验收要点：

- 支持可选 free-text instructions
- 可选支持上传 `.docx / .txt / .md` 领域资料
- 未提供领域知识时，系统使用默认停车经营分析准则

## 3. 功能需求

### FR-1：文件上传

系统必须支持上传：

- 报告模板 `.docx`
- 停车交易数据 `.csv`
- 可选领域知识文件
- 可选处理说明文本

系统应保存原始上传文件，并为每次请求创建唯一 `job_id`。

### FR-2：异步任务生命周期

系统必须支持：

- 创建 job
- 后台异步生成报告
- 查询 job 状态
- 下载生成结果

状态包括：

- `pending`
- `running`
- `completed`
- `failed`

### FR-3：CSV 数据清洗与校验

系统必须：

- 支持 UTF-8 BOM CSV
- 标准化列名
- 校验必需列
- 将金额字段解析为 Decimal
- 将时间字段解析为 datetime
- 在数据格式错误时将 job 标记为 `failed`

必需字段包括：

- `应收金额`
- `实收金额(元)`
- `免费金额(元)`
- `充值卡扣费(元)`
- `抵扣金额(元)`
- `抵扣时长(小时)`
- `实际抵扣额(元)`
- `支付方式`
- `支付渠道`
- `收费时间`
- `进车时间`

### FR-4：确定性指标计算

系统必须从 CSV 中确定性计算六个硬指标：

1. 总交易笔数
2. 应收总金额
3. 实收总金额
4. 实际抵扣总额
5. 实收率
6. 主要支付方式

这些指标必须由代码计算，不允许由 LLM 生成或修改。

### FR-5：数据分析 Profile

系统应生成结构化数据分析结果，包括：

- 支付方式分布
- 支付渠道分布
- 停车时长分布
- 入场时间分布
- 收费时间分布
- 异常/关注点候选

这些结果作为 Agent 的事实输入。

### FR-6：Agent 报告规划

系统必须包含一个报告规划 Agent。

Agent 负责：

- 基于确定性 facts 选择报告重点
- 选择至少一张图表
- 生成支付方式与渠道章节摘要
- 生成停车时长章节摘要
- 生成 2-3 条补充观察
- 生成 2-4 条结论与建议

Agent 不负责：

- 计算硬指标
- 修改硬指标
- 直接绘制图表
- 直接生成 Word 文件

### FR-7：图表生成

系统必须至少生成一张真实图表，并插入 Word 报告。

候选图表包括：

- 支付方式分布图
- 支付渠道分布图
- 停车时长分布图
- 入场时间分布图
- 收费时间分布图

图表由确定性代码生成。

### FR-8：Word 报告生成

系统必须生成 `.docx` 文件，包含：

- 报告信息
- 关键指标
- 支付方式与渠道
- 至少一张图表
- 停车时长分析
- 补充观察
- 结论与建议

生成报告应以用户上传的 `.docx` 模板为母版，而不是从空白 Word 重新创建。系统应尽量保留模板中的：

- 页面边距和基础版式
- 标题和章节样式
- 表格结构、边框和底色；模板说明列可在成品中删除
- 模板中的红色斜体占位样式
- 已存在的章节顺序

系统应在模板原位置替换或填充占位内容，例如：

- 将关键指标表格中的“填写值”列替换为确定性计算结果，并删除仅供参考的“示例预期值”列
- 将支付方式与渠道说明占位替换为 Agent 章节摘要
- 将图表占位替换为真实图表
- 将补充观察占位替换为 2-3 条 Agent observations
- 将结论与建议占位替换为 2-4 条 Agent recommendations

如果上传模板结构无法识别，系统可以回退到标准报告生成模式，但必须记录 fallback 原因。

### FR-9：结构化日志

系统必须记录：

- 用户请求输入信息
- job id
- job 生命周期事件
- LLM 调用信息
- 错误信息

日志格式为 JSON，输出到 stdout 或文件。

### FR-10：测试

系统必须包含至少一个 API-level test，覆盖：

```text
submit -> status -> download
```

报告生成过程在该测试中可以 mock。

## 4. 非功能需求

### NFR-1：正确性

六个硬指标必须可重复、准确，给定相同输入应得到相同结果。

### NFR-2：可观测性

系统必须能通过日志追踪：

- 用户提交了什么文件
- job 当前状态
- 任务在哪一步失败
- LLM 使用了什么 prompt
- LLM 返回了什么 response
- LLM 调用耗时

### NFR-3：可调试性

Agent 输出必须是结构化 `ReportPlan`，并带有 `source_fact_ids`。系统应能验证 Agent 的观察和建议来自已有 facts。

### NFR-4：安全性

系统不得：

- 将 API key 写入日志
- 将 `.env` 提交到 GitHub
- 在日志中输出完整原始 CSV
- 允许用户下载其他 job 的文件

v1 不实现用户认证，但文件访问必须通过 `job_id` 控制。

### NFR-5：可运行性

系统必须支持单命令启动：

```bash
docker compose up --build
```

项目应提供 `.env.example`，真实 LLM 模式通过 `OPENAI_API_KEY` 配置。

### NFR-6：可测试性

核心业务逻辑应可被单元测试或集成测试覆盖，包括：

- CSV 清洗
- 指标计算
- Agent 输出校验
- API 生命周期

### NFR-7：可扩展性

v1 使用本地文件存储和 DB-backed worker，但设计上应保留未来替换空间：

- 本地存储可替换为 S3
- DB worker 可迁移到 Temporal
- 领域上下文可扩展为 RAG/Qdrant

## 5. 验收标准

### AC-1：一键运行

在本地执行：

```bash
cp .env.example .env
docker compose up --build
```

系统可以启动 API、worker、数据库和前端。

### AC-2：提交任务

用户通过前端或 API 上传模板和 CSV 后，系统返回 `job_id`，HTTP 请求不会等待报告生成完成。

### AC-3：状态查询

用户可以通过 `job_id` 查询任务状态。刷新页面后仍能看到正确状态。

### AC-4：报告下载

任务完成后，用户可以下载 `.docx` 报告。

### AC-5：硬指标正确

报告中必须出现六个硬指标，且值来自 CSV 的确定性计算。

### AC-6：图表存在

报告中至少包含一张根据实际数据生成的图表，不允许只是模板占位图。

### AC-6.1：模板保真

生成报告应保留用户上传模板的主要视觉结构。以当前停车报告模板为例，验收时应确认：

- 标题和蓝色章节标题样式基本保留
- “关键指标”保留模板表格样式，成品仅展示“指标”和“填写值”两列
- 表头底色、表格边框和列结构基本保留
- 红色占位内容被实际数据或 Agent 内容替换
- 不应出现与模板完全不同的空白文档式报告

### AC-7：Agent 输出有效

报告中必须包含：

- 2-3 条补充观察
- 2-4 条建议
- 观察和建议应基于数据 facts
- 不得出现无依据的具体数字

### AC-8：日志完整

系统应能输出结构化 JSON 日志，至少包括：

- `job_created`
- `job_started`
- `metrics_computed`
- `llm_call_started`
- `llm_call_completed`
- `job_completed` 或 `job_failed`

### AC-9：测试通过

项目包含一个 API-level test，覆盖：

```text
submit -> status -> download
```

该测试不依赖真实 OpenAI API。

1. 建议定期复核优惠和抵扣核销规则。

3. 建议关注收费高峰时段的出口通行效率。

## 7. 关键设计决策

### 7.1 简单性

v1 不引入 Temporal、Celery、Qdrant、S3 或 Kubernetes。系统使用 FastAPI、PostgreSQL 和 Python worker 满足异步生成和状态持久化。

这样可以把实现重点放在：

- 指标正确性
- Agent grounding
- Word 报告质量
- 日志可观测性
- API 生命周期

### 7.2 自动化

系统自动完成：

- CSV 清洗
- 指标计算
- 数据 profile
- 经营关注点候选生成
- Agent 报告规划
- 图表生成
- Word 报告输出
- job 状态更新
- structured logging

用户只需要上传文件并下载结果。

### 7.3 灵活性

系统支持：

- 不同但相似结构的 Word 模板
- 基于模板占位符的格式保留和内容替换
- 可选处理说明
- 可选领域知识文件
- 可配置 OpenAI model
- 可配置 `OPENAI_BASE_URL`
- 无 key mock 模式

内部通过 `ReportFacts`、`DomainContextPack`、`ReportPlan` 等结构化接口降低模块耦合。

### 7.4 安全性

系统不会提交或暴露 API key。`.env` 不进入 GitHub，只提交 `.env.example`。

日志避免记录：

- API key
- 完整原始 CSV
- 敏感本地路径
- 不必要的用户原始文件全文

下载接口仅允许通过 job 关联的输出文件返回报告。

### 7.5 标准化

系统统一使用：

- REST API
- JSON 状态响应
- JSON structured logs
- Pydantic schema
- Decimal 金额计算
- `.docx` 标准输出
- Docker Compose 本地运行
- pytest 自动化测试

Agent 输出统一为 `ReportPlan`，不允许直接返回不可校验的自由文本。
